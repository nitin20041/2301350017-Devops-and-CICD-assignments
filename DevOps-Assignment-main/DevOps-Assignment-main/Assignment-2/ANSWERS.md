# Practice Questions - Written Answers

### Q1 Containers vs VMs: Explain the difference between a container and a virtual machine. In what scenario would you choose one over the other?
**Answer:** Containers virtualize the operating system, sharing the host kernel and running as isolated processes, making them lightweight and fast to start. Virtual Machines (VMs) virtualize the underlying hardware, including a full guest operating system, which provides stronger isolation but consumes more resources (CPU, RAM, disk). Choose containers for microservices, high-density deployments, and CI/CD where speed and portability are key. Choose VMs when you need strong security isolation (e.g., running untrusted code), different OS kernels, or legacy applications that require direct hardware access.

### Q2 Image layers: Your Docker image is 800 MB. A teammate suggests combining all RUN commands into one. Why would that help, and are there trade-offs?
**Answer:** Combining `RUN` commands reduces the number of layers in the image. Since each layer adds metadata overhead and can potentially store temporary files that aren't deleted until later layers, chaining commands (e.g., `apt update && apt install ... && rm -rf /var/lib/apt/lists/*`) ensures that temporary files are never committed to a layer, significantly reducing size. The trade-off is reduced cache efficiency; if one part of a giant `RUN` command changes, the entire command must be re-executed. It also makes the Dockerfile slightly harder to read and debug.

### Q3 Secrets management: A junior dev hardcoded a DB password in the Dockerfile and pushed to GitHub. Walk through exactly how you would remediate this — both the immediate fix and long-term prevention.
**Answer:** **Immediate fix:** 1. Rotate the compromised password immediately in the database. 2. Remove the secret from the code, replacing it with an environment variable reference. 3. Use tools like `BFG Repo-Cleaner` or `git filter-repo` to purge the secret from the entire Git history. **Long-term prevention:** 1. Implement a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager, or GitHub Secrets). 2. Add pre-commit hooks (like `gitleaks` or `trufflehog`) to scan for secrets before they are committed. 3. Educate the team on 12-factor app principles and secure DevOps practices.

### Q4 CI/CD failure: Your CD pipeline pushed a broken image to production at 2 AM. No one is awake. What automated safeguards could have prevented this, and what is your rollback strategy?
**Answer:** **Safeguards:** 1. Implement **Health Checks** in the deployment (like Kubernetes liveness/readiness probes or Render's healthCheckPath) to prevent traffic from hitting unhealthy containers. 2. Use **Canary Deployments** or **Blue/Green Deployments** to roll out changes to a small percentage of users first. 3. Add automated smoke tests in the CD pipeline that run against a staging/preview environment before production. **Rollback strategy:** Automated rollback triggered by health check failures, or a simple "one-click" manual rollback in the CI/CD UI to revert to the previous successful image tag (e.g., `latest` points back to the previous stable SHA).

### Q5 Stateless vs stateful: Your API is stateless and the DB is stateful. How does this distinction affect how you scale and deploy each one differently?
**Answer:** Stateless components (API) are easy to scale horizontally; you can spin up many identical replicas behind a load balancer because they don't store local data. They can be killed and restarted on any node without loss. Stateful components (DB) require careful management of persistent storage (Volumes) and usually follow a leader/follower (primary/replica) model for scaling. Scaling DBs often involves vertical scaling (bigger machine) or complex sharding/replication strategies. Deployments for stateful apps must ensure data consistency and avoid simultaneous restarts of all replicas.

### Q6 IaC philosophy: What does 'infrastructure as code' actually mean beyond just writing config files? What problems does it solve that clicking through a cloud console does not?
**Answer:** IaC is the practice of managing and provisioning infrastructure through machine-readable definition files rather than manual processes. It brings software engineering rigor to ops: version control, peer reviews (PRs), and automated testing. It solves the **"drift"** problem where manual changes make environments inconsistent. It enables **repeatability** (deploying the same stack in minutes), **visibility** (change history in Git), and **disaster recovery** (rebuilding the entire environment from scratch).

### Q7 Observability: Explain the difference between logs, metrics, and traces. If your API is slow, which would you check first and why?
**Answer:** **Logs** are discrete events (e.g., "User logged in"). **Metrics** are numerical aggregations over time (e.g., "Average latency is 200ms"). **Traces** track a single request across multiple services (e.g., "Request X spent 50ms in API and 150ms in DB"). If the API is slow, I'd check **Metrics** first (specifically Latency Histograms) to confirm the scope of the problem (is it all users or just one endpoint?). Once confirmed, I'd use **Traces** to find the bottleneck (e.g., a slow DB query) and **Logs** to see specific error messages or context for those slow requests.
